Kernel 8: Warptiling 93.7%

By kernel 7 we had squeezed the block-tiling design about as far as brute force could take it. We had a 128 × 128 block tile, an 8 × 8 register tile per thread, float4 vectorized loads, and an autotuner that had swept the whole parameter space and handed back the best combination it could find: 84.8% of cuBLAS. That is a genuinely good kernel. It is also stuck. The autotuner was optimizing a design that only knew about two levels of the hardware — the block and the thread — while the machine underneath it runs on three. This kernel adds the level we were ignoring: the warp.

The premise is simple to state and surprisingly hard to appreciate: on an NVIDIA GPU the block is not the unit of execution and the thread is not the unit of scheduling. The warp — 32 threads issued in lockstep — is what the hardware actually schedules, and everything from instruction issue to register-bank access to shared-memory conflicts is organized around it.1 A block is a software abstraction that gets carved into warps at launch: a 256-thread block is 8 warps, and those 8 warps are distributed across the SM's four warp schedulers. Nothing in the block-tiling kernels ever named a warp, so the mapping of work to schedulers was left to chance. Our block-tiled kernel had warps in it — it could not not have — but it never reasoned about them. Adjacent threads in a warp were grabbing whatever slice of the block tile the flat index handed them, which meant a single warp's 32 threads could be scattered across a wide, tall strip of the output. This kernel fixes that by inserting an explicit tiling level between block and thread, so the parallelism hierarchy in the code matches the parallelism hierarchy in the silicon exactly.

The three levels, made explicit

Here is the whole idea in one breath. A block tile is the chunk of C one thread block owns; it lives on one Streaming Multiprocessor (SM). A warp tile is the sub-chunk one warp owns; it lives on one of that SM's four warp schedulers. A thread tile is the little 8 × 8 patch of accumulators one thread owns; it lives in that thread's registers. Blocks parallelize across SMs, warps parallelize across schedulers within an SM, and threads parallelize across the ILP the scheduler can extract from a single warp. Three levels of tiling, three levels of hardware, one-to-one.

Concretely, keep the 128 × 128 block tile from kernel 7. Choose a block of 128 threads, which is 4 warps. Partition the block tile into a 2 × 2 grid of warp tiles, giving WM = 64 rows and WN = 64 columns per warp — one warp tile per warp, four of them, tiling the 128 × 128 block. Inside a warp tile, the 32 threads arrange themselves as an 8 × 4 grid, each thread owning an 8 × 8 register tile. Check the arithmetic: a 64 × 64 warp tile covered by 32 threads each doing 8 × 8 is (64 × 64) / (32 × 8 × 8) = 2. That leftover factor of two is not slack — it is a real degree of freedom we are about to name.

Warp sub-tiles: WNITER and WMITER

A warp does not have to cover its 64 × 64 region in one contiguous 8×4-thread stamp. It can stamp a smaller warp sub-tile several times, striding across the warp tile. That count is WNITER (sub-tiles along N) and WMITER (along M), and it is the factor of two we just found: with WMITER = 1 and WNITER = 2, each warp lays down two 64 × 32 sub-tiles side by side to cover its 64 × 64 region.2 Why bother splitting one warp tile into multiple sub-tiles instead of just making the thread tile bigger? Because the sub-tile stride interleaves each thread's outputs across the warp tile, which spreads a thread's float4 shared-memory reads across more banks and keeps the per-thread register tile at a size the register file can actually hold — 255 registers per thread is a hard ceiling. The sub-tile dimensions fall out as WSUBM = WM / WMITER and WSUBN = WN / WNITER. These four numbers — WM, WN, WMITER, WNITER — are the new knobs the autotuner gets to turn, and they are the entire content of the warp level.

The payoff is register reuse, and it is worth being precise about. Inside the k-loop, each thread loads a fragment of A from shared memory into registers (call it regM) and a fragment of B into registers (regN), then does an outer product regM ⊗ regN into its accumulators. The order of the loop nest decides how many times each loaded fragment gets used before it is overwritten. If we load regM once and reuse it across every column sub-tile, and load each regN once and reuse it across every row sub-tile, then a single shared-memory load feeds many FMAs. That is the register-cache locality the warp level buys us: fewer shared-memory transactions per flop, and each loaded operand amortized across the whole warp sub-tile grid.

The loop nest

Concept first, so the code reads as confirmation rather than puzzle. From outside in: iterate over the BK slice of the shared tile (dotIdx); for each slice, first load all the regM and regN fragments this thread needs across its WMITER and WNITER sub-tiles; then run the four innermost loops — over WMITER, WNITER, TM, TN — doing pure FMAs against registers with no memory touched. Separating the loads from the compute is the whole trick: it lets the compiler see a dense block of independent FMAs and schedule them back-to-back, and it lets each loaded fragment be reused across the sub-tile grid.

// registers for this thread's fragments and accumulators
float regM[WMITER * TM] = {0.0f};
float regN[WNITER * TN] = {0.0f};
float threadResults[WMITER * TM * WNITER * TN] = {0.0f};

for (uint dotIdx = 0; dotIdx < BK; ++dotIdx) {
    // 1. stage this thread's A/B fragments from SMEM into registers
    for (uint wSubM = 0; wSubM < WMITER; ++wSubM)
        for (uint i = 0; i < TM; ++i)
            regM[wSubM * TM + i] =
                sharedA[dotIdx * BM + warpRow * WM + wSubM * WSUBM
                        + threadRowInWarp * TM + i];
    for (uint wSubN = 0; wSubN < WNITER; ++wSubN)
        for (uint i = 0; i < TN; ++i)
            regN[wSubN * TN + i] =
                sharedB[dotIdx * BN + warpCol * WN + wSubN * WSUBN
                        + threadColInWarp * TN + i];

    // 2. outer products — registers only, no memory
    for (uint wSubM = 0; wSubM < WMITER; ++wSubM)
      for (uint wSubN = 0; wSubN < WNITER; ++wSubN)
        for (uint tm = 0; tm < TM; ++tm)
          for (uint tn = 0; tn < TN; ++tn)
            threadResults[(wSubM * TM + tm) * (WNITER * TN) + wSubN * TN + tn]
                += regM[wSubM * TM + tm] * regN[wSubN * TN + tn];
}

Nothing about the math changed from kernel 7 — same block tile, same float4 loads into shared memory, same accumulation. What changed is the indexing: warpRow and warpCol are derived from threadIdx.x / 32 (the warp's ID within the block), and threadRowInWarp/threadColInWarp from threadIdx.x % 32 (the lane within the warp). Every thread now knows which warp it belongs to and lays its outputs down inside that warp's tile, densely, instead of wherever a flat index scattered them.

The evidence

Point Nsight Compute at the two kernels back to back and the SASS tells a clean story. In kernel 7 the inner loop's shared-memory loads and its FFMA instructions are tangled together — a load, a couple of FMAs, another load — because the compiler could not prove which loads were reusable, so it kept re-issuing LDS instructions. In the warptiled kernel the disassembly separates cleanly into a burst of LDS instructions at the top of each dotIdx iteration followed by a long uninterrupted run of FFMA reading only registers.3 The mechanism the hardware provides here is the operand collector / register-cache stage between the register file and the FMA pipes. A tight, predictable per-warp register footprint with repeating strides hits in that stage and dodges register-bank conflicts; a scattered footprint does not. This is invisible in PTX and only shows up once you read SASS. The ratio of FFMA to LDS in the inner loop goes up, which is exactly "more flops per shared-memory byte" spelled in assembly.

There is a cost, and it is honest to name it: the explicit warp tiling adds registers. Each thread now holds regM, regN, and a fatter threadResults array, and the WMITER × WNITER sub-tile scheme multiplies the fragment count. With a 256 KB register file per SM (65536 × 32-bit registers) and a 255-register-per-thread ceiling, register pressure — not shared memory — becomes the thing that caps occupancy. The autotuner's job at this level is essentially to trade sub-tile count against register footprint until occupancy and reuse balance. That balance is why WNITER matters as much as it does: it is the dial between "wide reuse, many registers" and "narrow reuse, more resident warps."

Autotune the six parameters — BM, BN, BK, WM, WN, WNITER (with TM, TN following) — and the warptiled kernel lands at 93.7% of cuBLAS. That is the number. From 84.8% to 93.7% is not a dramatic multiple; it is the last hard nine you buy by making the scheduler's job trivial instead of merely possible. We are now inside cuBLAS's own territory, close enough that the remaining gap is no longer about tiling at all.

Where the last 6% lives

It is tempting to keep pushing this design — a tenth warp knob, a cleverer swizzle, one more pass of the autotuner. It will not get us the remaining 6.3%, and it is important to understand why. Every kernel on this ladder so far, including this one, has been a CUDA-core kernel: it does its multiply-adds on the FP32 FMA pipes. cuBLAS, on a real workload, does not — it issues to the tensor cores, which do a whole matrix-multiply-accumulate per instruction and deliver an order of magnitude more throughput than the FMA pipes ever can. Getting the last stretch means changing the instruction, not the tiling.

But the tiling is not wasted — it is the prerequisite. Tensor-core instructions consume operands in fixed fragment shapes, fed from shared memory, computed by warps. The block→warp→thread hierarchy we just built is precisely the addressing you need to stage those fragments: a warp is exactly the granularity a tensor-core instruction operates at. So the mental model transfers directly. In the next kernels we keep this three-level skeleton and swap the innermost FFMA loop for the tensor-core path — mma on Ampere, and on this H100 the asynchronous, warpgroup-wide wgmma that eats a 64 × N fragment straight out of shared memory. That is where the remaining nine lives, and it is the subject of the next section.
