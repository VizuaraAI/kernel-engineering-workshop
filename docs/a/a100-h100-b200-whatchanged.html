01 · The GPU, From Silicon Up

A100 → H100 → B200: what changed

Every time NVIDIA ships a new datacenter GPU, a cohort of kernel engineers has to relearn part of their job. Not because the old tricks stop working — a well-coalesced, well-tiled GEMM from 2020 still runs on a 2024 chip — but because the fastest way to feed the math units changes, and the old kernel now leaves most of the silicon idle. If you want to understand why the GEMM ladder we climb elsewhere on this site keeps having to grow new rungs, you have to understand what physically changed between Ampere, Hopper, and Blackwell.

There is one thread that ties all three generations together, and it is the single most important fact in this article: each generation adds FLOPs faster than it adds bandwidth. The tensor cores get much wider; HBM gets a little faster. From the three regimes we know what that does to the roofline — it pushes the ridge point (the arithmetic intensity above which you can be compute-bound) steadily to the right. So each new chip is, in effect, hungrier: it demands that you reuse every byte more aggressively than the last one did, or the wider math units starve. Almost every architectural feature below exists to solve exactly that starvation problem.

A100 (Ampere): the asynchronous copy arrives

The A100 is where the modern GEMM kernel starts to look modern. Two things it introduced still shape how we write code today.

The first is cp.async. Before Ampere, staging a tile of A and B into shared memory meant a two-hop journey: a thread issued a global load into a register, then wrote that register into shared memory. Every byte you tiled paid a tax in register pressure and, worse, the load blocked the thread — the SM had nothing to do but wait on HBM (High Bandwidth Memory) latency of hundreds of cycles. cp.async broke that dependency. It copies global memory straight to shared memory without touching registers, asynchronously, so the thread can issue the copy for the next tile and immediately go compute on the current one.1 PTX spells it cp.async.cg.shared.global; the .cg variant bypasses L1 and caches only in L2, which is usually what you want for streaming tiles you will not re-read from global. .ca caches in L1 as well. That is software pipelining — double buffering — and it is the reason a good Ampere GEMM overlaps memory and math instead of alternating between them.

The second is TF32 (TensorFloat-32), a 19-bit tensor-core mode with FP32's 8-bit exponent but only 10 mantissa bits. It let training code get an ~8× tensor-core speedup over true FP32 by quietly truncating the mantissa, with a precision loss most networks never notice. TF32 is the moment the tensor cores stopped being a special-case feature for inference and became the default path for the bulk of deep-learning math.

Both features answer the same pressure. Ampere's BF16 tensor throughput (about 312 TFLOP/s on the A100) was far ahead of its ~2 TB/s of HBM2e, so the chip could only stay fed if the copy engine ran ahead of the math. cp.async is the tool that lets it.

H100 (Hopper): the tile becomes a first-class object

Hopper is the largest single jump in how you write a GEMM, because it stopped treating the tile as something threads assemble by hand and made it a hardware primitive. Four features matter, and they interlock.

TMA (Tensor Memory Accelerator) replaces cp.async for the heavy lifting. Instead of every thread computing an address and issuing its own copy, you build a 128-byte tensor map descriptor once on the host — it encodes the matrix's shape, strides, and tile geometry — and then a single thread issues one bulk copy of an entire 2D (or higher-dimensional) tile from global to shared memory.2 The instruction is cp.async.bulk.tensor.2d.shared::cluster.global; the rest of the block waits on an mbarrier that TMA arrives on when the transfer completes. One thread launches it, the hardware does the address generation. TMA also does swizzling in hardware — it lays the tile into shared memory in a bank-conflict-free pattern automatically. That single feature deletes an entire category of kernel code: the manual padding tricks (adding 4 elements to a stride of 128 so accesses spread across the 32 shared-memory banks) that earlier kernels needed to avoid serialization now happen for free.

wgmma (warpgroup MMA) is the new math instruction, and it is enormous. A single wgmma.mma_async.sync.aligned.m64n64k16 issues 65,536 multiply-accumulates (64 × 64 × 16) across a warpgroup — four warps, 128 threads — in one instruction. Compare that to the scalar FFMA in a naive kernel, which retires one MAC per instruction issue: wgmma collapses roughly 65,000 issues into one, which is why the instruction decoder stops being the bottleneck. The 64×64 output tile lives distributed across the warpgroup's registers, 32 accumulators per thread in a hardware-fixed layout. Crucially it is mma_async: the instruction queues and returns, and you synchronize later with wgmma.commit_group / wgmma.wait_group. So the math itself now pipelines the same way TMA copies do.

Thread-block clusters and DSMEM (Distributed Shared Memory) are the third piece. A cluster is a group of up to 16 thread blocks that the scheduler guarantees run concurrently on SMs within one GPC (Graphics Processing Cluster) — and the H100 has 8 GPCs across its 132 Streaming Multiprocessors (SMs), 18 per GPC.3 16 is the non-portable maximum; the cluster size CUDA guarantees on any architecture is 8. A cluster maps onto a single GPC, which is why the ceiling tracks GPC size rather than total SM count. Blocks in a cluster can read each other's shared memory directly, over a dedicated SM-to-SM network, without a round-trip through L2 (~50 MiB, split across two partitions and a crossbar). DSMEM sits in the latency gap between per-SM shared memory (~20–30 cycles) and L2. For GEMM this means a tile fetched once by one SM can feed several — you multiply the effective reuse of every expensive HBM read across the cluster instead of re-fetching per block.

FP8 is the fourth. Hopper's tensor cores natively multiply e4m3 and e5m2 8-bit floats, doubling throughput over BF16 and halving the bytes you move. But 8 bits of dynamic range is thin, so FP8 GEMM needs per-tensor or per-block scaling factors to keep values in range — the first taste of the "quantize, then rescale" dance that Blackwell makes central. Because Hopper packs SMEM and L1 into 256 KiB per SM (up to 228 KiB usable as shared memory), you finally have enough on-chip space to stage the deep, multi-stage software pipelines that TMA and wgmma want to be fed.

Put together, a Hopper GEMM reaches for a completely different toolbox than an Ampere one. The naive-to-tuned ladder still applies — you still fix coalescing, still tile into shared memory, still climb from 1.3% of cuBLAS toward the high nineties — but the top rungs are now TMA + wgmma + cluster-level reuse rather than hand-written float4 loads and register tiles.

B200 / GB300 (Blackwell): the accumulator leaves the registers

Blackwell pushes the same logic one step further, and the step is subtle enough that it trips up engineers porting Hopper kernels. On Hopper, the wgmma output tile lives in registers — that is a hard ceiling, because the register file is only 256 KB per SM (65536 32-bit registers, max 255 per thread), and the accumulator competes with everything else for it. As tensor cores got wider still, that ceiling started to bind.

TMEM (Tensor Memory) is Blackwell's answer: a new, dedicated on-chip memory bank that holds tensor-core accumulators, physically separate from the register file. The new MMA instruction, tcgen05, reads its operands from shared memory and writes its result into TMEM, not into registers.4 This is why a naive Hopper→Blackwell port can actually get slower before it gets faster: the accumulator data path moved, and code that manually shuffles wgmma register fragments now has to be rewritten to stage results through TMEM and copy them out explicitly. Freeing the register file lets each thread do more bookkeeping — addressing, scaling, pipelining — around a math instruction that is now even larger.

CTA pairs are the second change. Where Hopper's cluster let blocks share memory, Blackwell lets two CTAs (Cooperative Thread Arrays — thread blocks) on adjacent SMs jointly issue a single MMA that spans both, a "2-CTA MMA." The pair cooperates on one larger tile, reading operands and writing accumulators across the CTA-pair boundary. It is the natural extension of the tile-as-hardware-object idea: first the tile outgrew the thread (wgmma), then it outgrew the warpgroup, and now the widest tiles outgrow a single block.

NVFP4 is the headline, and it is where the "add FLOPs faster than bandwidth" story reaches its logical end: if you cannot move more bytes, make each byte carry more math. NVFP4 is a 4-bit float, e2m1 — 2 exponent bits, 1 mantissa bit — which is almost no numeric range at all on its own. It works because of micro-block scaling: every group of 16 FP4 elements shares one FP8 (e4m3) scale factor.5 Contrast the coarser MXFP4 convention of 32-element blocks with an e8m0 power-of-two scale; NVFP4's 16-element blocks and richer e4m3 scale give it meaningfully better accuracy per bit, which is why it is the format Blackwell's tensor cores decode in hardware. So the data path is: load packed 4-bit weights and their FP8 scales, and let the hardware do the dequantization inline. Blackwell provides PTX conversion intrinsics — cvt.rn.f16x2.e2m1x2 unpacks two FP4 values to FP16 at once — that cut the decode instruction count by roughly 10× versus doing the bit-twiddling in software.

The arithmetic that motivates all of this is stark. A B200 does on the order of 2.2 PFLOP/s of dense FP8 tensor math and more again in NVFP4, against roughly 8 TB/s of HBM3e. The compute climbed far more than the bandwidth did, exactly as it has every generation — so the ridge point sits further right than ever, and a GEMM that does not exploit 4-bit weights, inline dequant, and TMEM-staged accumulation simply cannot keep those tensor cores lit.

What this means for the ladder

Step back and the three generations tell one continuous story. Ampere gave us the asynchronous copy so math and memory could overlap. Hopper made the tile a hardware object — copied by TMA, consumed by wgmma, shared across a cluster — and added FP8 to move fewer bytes. Blackwell moved the accumulator out of the register file into TMEM, paired CTAs for still-larger tiles, and pushed precision down to 4 bits with hardware block-scaling. Each move is the same move: the tensor cores got wider than the memory system, and the architecture responded by finding one more way to reuse a byte or shrink a byte.

That is why the GEMM ladder is never finished. The naive kernel at 1.3% of cuBLAS and the coalescing, shared-memory, and register-tiling rungs that carry you to the high-nineties on one generation are table stakes on the next — necessary, but no longer sufficient. On Ampere the top of the ladder was vectorized loads and warp tiling; on Hopper it grew a TMA-and-wgmma rung; on Blackwell it grows a TMEM-and-NVFP4 rung above that. When you profile a kernel and find it far below the roofline, the first question is still the one from the three regimes — what is it waiting on? — but the answer's remedy now depends on which silicon you are standing on. Knowing what changed between these three chips is knowing which rung to reach for.
