03 · The GEMM Worklog

Tensor cores II: fragments & swizzling TC

In the previous kernel we finally lit the tensor cores: we issued a real mma instruction and watched the throughput jump. But we left a splinter under the fingernail. The profiler reported that our shared-memory loads on the tensor-core feed path were running at roughly eight times the ideal number of wavefronts — an 8-way bank conflict, hiding in plain sight, throttling every single MMA we issue. This kernel is about pulling that splinter out, and the tool for the job is one of the strangest-looking tricks in the whole ladder: we are going to deliberately scramble the addresses we write to shared memory so that they come back out cleanly.

To get there, we first have to understand something we quietly skipped: on a tensor core, the warp is the unit of work, and the operands do not live where you think they live. So before the swizzle, the fragments.

Who holds what: the register-fragment layout

A CUDA core takes scalar operands from a single thread's registers. A tensor core does not. It consumes a small dense matrix multiply — for the shape we are using, mma.sync.aligned.m16n8k8 — as a collective operation across all 32 threads of a warp.1 We use the m16n8k8 shape (16×8 output, K=8) that the tensor-core reference builds on. There are wider shapes (m16n8k16) that halve the instruction count along the contraction dimension, but they do not change the fragment-layout or bank-conflict story; the win we are after is the same. The 32 threads pool their registers, the hardware reads that pool as three matrices, multiplies, and writes the result back into the same pooled registers. No thread ever holds a full row or column. Each thread holds a scattered handful of elements, and the map from (thread, register) to (row, col) is fixed by the ISA — you do not get to choose it.

Concretely, for m16n8k8 in FP16 with an FP32 accumulator, the A fragment (16×8) is distributed so that each thread carries four half values packed into two 32-bit registers; the B fragment (8×8) gives each thread two half values in a single register; and the C/D accumulator (16×8) gives each thread four float values in four registers. The layout is not row-major and it is not column-major — it is an interleaved, quadrant-based pattern the tensor core was wired to expect.

This has a sharp practical consequence. We stage tiles of A and B in shared memory, exactly as in the shared-memory kernel. But we can no longer let each thread copy "its" row out of shared memory with a plain indexed load, because the tensor core wants that quadrant-interleaved arrangement, not a contiguous row. Rearranging a naturally-stored tile into fragment layout with ordinary loads would take a storm of scalar instructions and shuffles.

So Ampere (and Hopper after it) gives us a dedicated instruction: ldmatrix. One ldmatrix.sync issued by the warp reads one to four 8×8 patches of FP16 out of shared memory and deposits them directly into the 32 threads' registers already in fragment layout, doing all the cross-thread shuffling in hardware. It takes one shared-memory address per row — thread i supplies the base address of row i — and gathers all eight rows of an 8×8 tile in one shot. It is the bridge between "how we store tiles" and "how the tensor core reads them," and it is exactly on this bridge that the bank conflict lives.

The bank conflict, made concrete

Shared memory on an H100 is one of the 256 KiB the SM splits between L1 and scratchpad — up to 228 KiB of it usable as pure SMEM — organized into 32 banks, each serving 4 bytes per cycle.2 Those 228 KiB are not exactly 228; the split is configured in fixed steps and a sliver is reserved. The bank count and 4-byte word, however, are exact and have been stable across many generations. A warp's 32 lanes can hit all 32 banks in one cycle if and only if their addresses land in 32 distinct banks. If two lanes want two different 4-byte words that map to the same bank, the hardware serializes them: a 2-way conflict costs two cycles, an 8-way conflict costs eight.

Now look at what ldmatrix asks for. To assemble one A fragment, the warp supplies the addresses of the 8 rows of an 8×8 half-precision tile. But that tile does not sit in isolation — it lives inside a much wider SMEM tile we staged from global, say 64 elements wide, stored row-major. So consecutive rows of our 8×8 sub-tile are 64 × 2 bytes = 128 bytes apart in shared memory. A bank spans 4 bytes and there are 32 of them, so the bank pattern repeats every 32 × 4 = 128 bytes — exactly the row stride. Every row of the sub-tile therefore begins at the same bank offset, and each row's eight half values (16 bytes) span the same four banks. Rows 0 through 7 all land in the same four banks. Eight rows, funneled into four banks: that is the 8-way conflict the profiler flagged, and it fires on every ldmatrix on the critical path.

The classic fix is padding: store each 16-wide row as 16 + 8 and let the extra columns push successive rows into different banks. It works, but it wastes shared memory — a scarce resource we want every byte of for bigger tiles — and it does not compose well once tiles get wide. The better fix costs zero bytes.

The swizzle: XOR the column with the row

The insight is that a bank is chosen by the low bits of the address, and XOR is a permutation. If, before we store an element, we perturb its column index by XOR-ing in some bits of its row index, we shuffle each row's elements into a different set of banks — without moving any element to a different row, and without using one extra byte. Reads and writes both apply the same permutation, so the data comes back correct; only the bank assignment changes.

The permutation that clears our conflict is, for a tile whose swizzle unit is the natural width, of the form:

// logical (row, col) -> swizzled column within the SMEM tile
__device__ __forceinline__
uint swizzle(uint row, uint col) {
    // XOR the column's low bits with a slice of the row bits.
    // Same permutation on store and on ldmatrix-address computation.
    return col ^ ((row & 0b1100) >> 2);
}

Written against the flat offset the way the reference does it, the same idea reads f(i) = i ^ ((i & 0b1100) >> 2) — XOR-ing two row bits into the bank-selecting field of the index.3 The exact mask depends on the tile's element size and width; the invariant is that the XORed bits must land in the address field that selects the bank (bits 2–6 for a 4-byte word across 32 banks) and must be a permutation within each row so no element is lost or aliased. Because XOR is its own inverse and touches only intra-row column bits, the store and the ldmatrix address computation stay in perfect agreement — we do not have to "un-swizzle" on the way out. Row 0 is untouched, the next row's elements shift by one bank-group, the row after that by two, and so on, so the eight rows that used to pile into four banks now spread across distinct banks.

The code change is almost insultingly small: everywhere we compute a shared-memory address — the float4 stores when we load tiles from global, and the per-row addresses we hand to ldmatrix — we route the column through swizzle(). No new buffers, no padding, no extra instructions in the hot loop beyond a single XOR that the compiler folds into the address arithmetic.

The profile: the conflict is gone

Predict-then-measure. The hypothesis is specific and falsifiable: the swizzle should take l1tex__data_bank_conflicts_pipe_lsu_mem_shared on the LDSM/ldmatrix instructions from an 8-way conflict to zero, which means the ratio of actual to ideal shared wavefronts should drop from roughly 8× to 1×.

That is exactly what Nsight Compute shows. Before the swizzle, the source view attributes a mountain of shared-load wavefronts to the ldmatrix lines, and the "Memory Workload Analysis" section reports the shared pipe as the top stall reason. After the swizzle, the bank-conflict counter on those same instructions reads 0, the actual-to-ideal wavefront ratio sits at 1.0, and the stall migrates off the shared-memory pipe entirely.4 A residual near-1.something ratio is normal and comes from the tail of tiles that don't fill a full warp's worth of rows; the thing you are watching for is the collapse of the 8× factor, not a literal 1.000. The tensor cores stop waiting on the scratchpad and start eating operands at the rate ldmatrix can actually supply them.

The number

With the load path unblocked, every MMA that was previously stalling on shared memory now issues back-to-back, and the warp's tensor throughput roughly doubles. On the 8192 × 8192 benchmark this kernel lands at about 50% of cuBLAS — up from roughly a quarter where the conflict-throttled version sat, a clean ~2× jump from removing a single structural stall.5 The exact multiplier depends on how memory-bound the pre-swizzle kernel was; the more of your runtime was hidden behind the 8× conflict, the bigger the win. In our config the conflict was the dominant stall, which is why one permutation moves the number this far. Half of cuBLAS from a permutation that costs zero bytes and one XOR is, pound for pound, one of the best trades on the entire ladder — comparable to the coalescing fix in the FP32 GEMM worklog, where a one-line reassignment of threads to elements sharply lifted throughput.

It is worth pausing on why this class of win exists at all. We are still, by the logic of the three regimes, fighting a memory-movement battle — just one level up the hierarchy. The naive kernel drowned in HBM traffic; the tiled kernel moved that pressure into shared memory; and now the shared-memory bank structure is the wall. Each optimization does not eliminate the bottleneck so much as promote it to the next-fastest memory. The swizzle is the move that finally lets shared memory feed the tensor cores at full rate.

Bridge to the next step

We are at 50% of cuBLAS and, for the first time, the tensor cores are no longer starved by the scratchpad. So where does the next stall come from? It comes from the fact that our ldmatrix and our mma still take turns: the warp loads a fragment, then multiplies it, then loads the next, and while it is loading, the tensor cores are idle. The load and the math are serialized when they could overlap.

The fix is to hide the shared-memory loads behind the math with a software pipeline — issue the ldmatrix for tile k+1 while the mma for tile k is still running — and, on Hopper, to feed that pipeline from HBM with the Tensor Memory Accelerator (TMA) doing bulk asynchronous copies. That is the next kernel, where we stop thinking about single instructions and start thinking about the timeline, and it is what carries us from 50% toward the low-70s and beyond.
