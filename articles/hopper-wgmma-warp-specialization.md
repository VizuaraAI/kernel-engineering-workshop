By the end of the [GEMM ladder](gemm-kernel-1-naive.html) we had reached **93.7% of cuBLAS** on FP32, and every trick that got us there was a variation on a single theme: keep the math units fed. Stage tiles in [shared memory](shared-memory-l1.html), reuse them across a block, vectorize the loads, tune the tile shape until the arithmetic units almost never stall. That whole ladder, though, was fighting with one hand tied behind its back. It used the CUDA cores — the scalar FMA pipeline — and never once touched the tensor cores, the units that hold essentially all of the H100's advertised **989 TFLOP/s** of BF16 throughput. This article is where we finally reach for them, and it turns out the tensor cores demand a completely different programming model than the one we spent ten kernels perfecting.

The reason is bandwidth of a different kind: instruction bandwidth. A scalar `FFMA` does one multiply-accumulate per instruction issue. To pull a real fraction of 989 TFLOP/s out of the chip, you cannot afford one issue per MAC — the warp scheduler would be the bottleneck long before the ALUs were. Hopper's answer is an instruction so large that a single issue does the work of tens of thousands of scalar FMAs, and a programming model built around keeping that instruction's operands arriving on time. That instruction is **Warpgroup Matrix Multiply-Accumulate** (WGMMA), and the model is warp specialization.

## One instruction, four warps

The scalar `FFMA` is issued per-thread: each of the 32 lanes in a **warp** does its own multiply-add. Hopper's `wgmma.mma_async` is issued per *warpgroup*. A **warpgroup** is exactly four contiguous warps — `128` threads — that cooperate on one matrix multiply.[[sn: The four warps must be the *aligned* group: warps 0–3, 4–7, and so on. `wgmma.mma_async.sync.aligned` will misbehave if you hand it four warps that do not form an aligned quad. The `.aligned` in the mnemonic is a promise you are making to the hardware, not a request.]] The canonical shape is written into the instruction name: `wgmma.mma_async.sync.aligned.m64nNk16.f32.bf16.bf16`. That reads as: multiply a `64 × 16` tile of `A` by a `16 × N` tile of `B`, accumulate into a `64 × N` tile of `C` in FP32, with BF16 inputs. `N` is chosen from a fixed menu — 16, 32, 64, up to 256 — and a single `m64n64k16` issue is on the order of `2 × 64 × 64 × 16 ≈ 131,000` floating-point operations from one instruction. That is the leverage we came for.

Where does that `64 × N` output tile live? In registers, spread across the whole warpgroup. This is the first thing that breaks your CUDA-core intuition. The register file is `256 KB` per **Streaming Multiprocessor** (SM) — `65536` 32-bit registers — but a single thread caps out at `255` registers. A `64 × 64` FP32 accumulator is `4096` values; there is no way one thread holds it. So WGMMA distributes it: each of the `128` threads owns a fixed fragment of the output — for `m64n64k16` that is `32` accumulator registers per thread, laid out in the exact `d[4][8]` pattern the PTX spec dictates.[[sn: You do not get to choose the fragment layout. Which output element lands in which thread's register is fixed by PTX §9.7.15; if you want to write `C` back correctly, or apply an epilogue, you have to follow that mapping element-for-element. This is the tax for the instruction being cheap.]] The accumulator never leaves registers across the whole `K` loop, and a flag on the instruction chooses whether this issue computes `C = A·B` or `C = A·B + C`, so accumulation across `K`-tiles costs nothing extra — no re-zeroing, no reload.

## Async from shared memory

The second surprise is where the *inputs* come from. On the CUDA-core ladder, we hand-loaded `A` and `B` fragments from shared memory into registers, then issued the FMAs. WGMMA skips the register staging entirely: the `A` and `B` operands are read **directly out of shared memory** by the tensor core, described not by pointers but by a packed `64`-bit **matrix descriptor** that encodes the SMEM base address, the leading-dimension stride, and the swizzle mode — all in `16`-byte units.[[sn: You build that descriptor with `__cvta_generic_to_shared()` to get the real SMEM address, then pack byte offsets shifted right by 4. The top two bits carry the swizzle mode; get them wrong and the tensor core silently reads a transposed or scrambled tile, which is a miserable bug to find because the kernel runs and just produces garbage.]]

And it is *async*. That is the whole point of the `_async` suffix. When a warpgroup issues `wgmma.mma_async`, the instruction returns almost immediately; the tensor core goes off and does the multiply in the background while the warpgroup's threads run ahead. You batch several issues, then bracket them:

```cpp
wgmma_fence();                  // publish SMEM writes to the tensor core
#pragma unroll
for (int n = 0; n < N_TILES; ++n)
    wgmma_m64n64k16(desc_a[n], desc_b[n], acc, /*accumulate=*/n != 0);
wgmma_commit_group();           // seal this batch of async MMAs
wgmma_wait_group<0>();          // stall until they retire, acc now valid
```

The `wgmma.fence.sync.aligned` before the batch is doing something subtle: it tells the tensor core that all prior writes into the SMEM tiles are complete and visible, so it is safe to start reading them. Then `commit_group` seals the batch and `wait_group<0>` blocks until every MMA in it has retired and the accumulator registers are valid. Between commit and wait is exactly the window the warpgroup can spend doing something *else* — and that "something else" is the entire game.

## The problem: who loads the next tile?

Here is the honest failure that motivates everything that follows. Write the obvious thing — one warpgroup that, in a loop, loads the next `K`-tile into SMEM, then issues WGMMA on it, then loads the next, then WGMMAs — and profile it. The tensor cores are idle most of the time. They finish a batch in a flash and then sit there while the *same warps* trudge through the next tile's load. We have recreated the naive-kernel disease one level up: the compute unit waits on the memory unit because the same threads are responsible for both, and they can only do one thing at a time.

The classic fix from the CUDA-core world is double buffering: prefetch tile `N+1` while computing on tile `N`. That still works, but Hopper gives us a cleaner separation of duties, and it starts with the load engine. The **Tensor Memory Accelerator** (TMA) is a dedicated copy engine that performs a bulk 2D tile transfer from global memory into shared memory, issued by a *single thread*, described by a `128`-byte tensor map built on the host with `cuTensorMapEncodeTiled()`.[[sn: TMA also does the swizzling for you. The 128-byte-swizzled SMEM layout that WGMMA's descriptor expects is exactly what `CU_TENSOR_MAP_SWIZZLE_128B` writes, so TMA and WGMMA are designed as a matched pair — one produces the layout the other consumes. Hand-rolling that swizzle is possible and documented nowhere pleasant.]] One thread fires the TMA, the copy runs asynchronously in the background, and a barrier signals when the bytes have landed. Now the load does not occupy 128 threads of compute — it occupies a copy engine. Which raises the design question: if loads and computes are done by different hardware, why are we still doing them from the same warps?

## Warp specialization: producers and consumers

So we split the warps by *role*. This is **warp specialization**. Inside one thread block we launch, say, one **producer** warpgroup and one or two **consumer** warpgroups, and they run completely different code:

- The **producer** warps do nothing but issue TMA loads. Their loop: wait until a slot in a shared-memory ring buffer is free, fire a TMA to fill it with the next `K`-tile of `A` and `B`, signal that the slot is full, advance. They never touch a tensor core.
- The **consumer** warps do nothing but WGMMA. Their loop: wait until the next ring-buffer slot is full, issue the `wgmma.mma_async` batch on it, signal that the slot is now free for the producer to refill, accumulate into their register-resident `C`.

The two roles rendezvous through a **shared-memory ring buffer** of `K`-tiles — a **software pipeline** with, typically, three or four stages of depth. The producer runs ahead, filling stages 2, 3, 4 while the consumer is still grinding on stage 1. As long as the producer stays ahead, the consumer's `wgmma.wait_group` never actually stalls: the next tile is always already in SMEM by the time the tensor core wants it. The tensor cores saturate.

## mbarriers: the handshake

The producer and consumer never call `__syncthreads()` with each other — they are not even the same warps, and a block-wide barrier would defeat the whole overlap. They coordinate through **mbarriers**, the shared-memory barrier objects Hopper exposes (`cuda::barrier` at block scope, or the raw PTX `mbarrier` API underneath). Each ring slot gets two: a *full* barrier the producer arrives on once its TMA completes, and an *empty* barrier the consumer arrives on once it has finished reading the slot. The producer waits on *empty* before overwriting a slot; the consumer waits on *full* before reading one. It is the textbook bounded-buffer handshake, implemented in silicon.

The detail that matters for performance is how *cheap* the arrival is. TMA can be told to signal an mbarrier directly on completion, so the producer's "the tile has landed" signal is a single hardware event, not `128` threads each calling `arrive()`. Dropping down to the raw PTX `mbarrier` intrinsics instead of the higher-level C++ `cuda::barrier` token API is worth it here: it collapses the per-tile synchronization from one arrival *per thread* down to a single arrival for the whole group. The raw `mbarrier.arrive` / `mbarrier.try_wait` path lets one thread arrive on behalf of everyone and skips the token bookkeeping the C++ API carries — which, on a tight `K`-loop, is a real and measurable slice of the runtime.

There is one more knob that makes specialization pay off: **per-warpgroup register allocation**. The consumers need a huge register budget — remember, the entire `C` accumulator is register-resident, so a consumer warpgroup wants something like `240` registers per thread. The producers issue TMAs and touch almost nothing; give them `24`. Hopper lets you *reallocate* registers between warpgroups at runtime (`setmaxnreg`), donating the register file to the warps that actually need it. Without this, a naive uniform allocation forces register spills on the consumers and the whole thing collapses.[[sn: This is why a common layout is *two* consumer warpgroups sharing *one* producer: it balances the register pressure and keeps all four tensor cores on the SM busy. One producer can comfortably outrun two consumers because TMA bandwidth, not issue rate, is what it is limited by.]]

## Where this lands

Stack it all up — WGMMA for the math, TMA for the loads, a multi-stage SMEM ring buffer, mbarrier handshakes, register reallocation — and the tensor cores go from "idle most of the time" to "essentially always busy". On `4096 × 4096` BF16, a clean producer/consumer WGMMA kernel gets you into the **~500 TFLOP/s** neighborhood, roughly half of the chip's realistic peak, before you have even started on the cluster-level and scheduling tricks. That is already a different universe from the CUDA-core ladder, which topped out around `40` TFLOP/s in FP32. The tensor cores were always there; the CUDA-core kernels just could not talk to them.

We are not at cuBLAS yet. The gap from here is not about the inner loop anymore — it is about *feeding the SMs from HBM and L2* efficiently across the whole `132`-SM chip: thread-block clusters and distributed shared memory so neighbouring SMs share tiles, TMA multicast so one HBM read fills several SMs at once, and block-scheduling that respects the `~50 MiB` L2's two-partition geometry so tiles stay resident. Those are the moves that carry a well-pipelined WGMMA kernel from half-peak past **100% of cuBLAS**. But none of them matter until the tensor cores are saturated — and that saturation is exactly what warp specialization buys.
