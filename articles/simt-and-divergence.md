The first time I profiled a kernel of mine and watched a single `if` cut the throughput in half, I assumed I had a bug. I did not — I had a warp. When I launch a CUDA kernel I think I am launching threads, but the hardware does not schedule 1024 independent little programs; it schedules them in bundles of exactly 32, and every thread in a bundle is dragged through the same instruction at the same time. That bundle is a **warp**, and the moment I understood that one instruction fetch is shared across 32 threads, half the mysterious performance cliffs in my GPU code stopped being mysterious.

This is the model NVIDIA calls **SIMT** — Single Instruction, Multiple Threads. It is why I can write ordinary scalar-looking code (`if`, `for`, `c = a + b`) and have it run on a machine that is really a very wide vector processor in disguise. It is also why a single innocent `if` can quietly halve throughput. My goal here is to work through how that happens and how to see it coming before I profile — no GEMM optimization yet, just the execution model I'll lean on for the rest of the ladder.

## One program counter, thirty-two threads

I start from the hardware. A **Streaming Multiprocessor** (SM) on the H100 is divided into four **processing blocks** (SM sub-partitions), each with its own warp scheduler and execution units.[[sn: Each of the four sub-partitions has its own register file slice, its own dispatch unit, and — on Hopper — its own set of FP32/INT32 pipes and a fourth-generation tensor core. A warp is scheduled onto exactly one sub-partition, never split across them.]] A warp scheduler picks one *ready* warp per cycle and issues one instruction from it. That instruction is then executed by 32 lanes in lockstep — lane `i` operates on the register state of thread `i`.

The consequence I keep coming back to: **the 32 threads of a warp share one program counter.** There is one instruction stream, one fetch, one decode, and 32 sets of registers and 32 sets of data flowing through it. When my code says `threadIdx.x`, every lane reads its own lane index; when it says `acc += A[i] * B[i]`, all 32 lanes do a multiply-add in the same cycle on their own operands. This is why GPUs are so cheap per FLOP: the expensive parts of a CPU core — fetch, decode, scheduling, branch prediction — are amortized across 32 threads instead of one. Warp size has been `32` on every NVIDIA architecture since Tesla; it is not guaranteed by the model — I read `warpSize` when I want to be pedantic — but no shipping GPU has ever used another value.

## What a branch actually costs

Now the trouble. My hypothesis, before I ever touch the profiler, is that a branch inside a warp cannot be free — one program counter cannot point at two instructions. Say my source has a branch:

```cpp
if (threadIdx.x < 16)
    x = expensive_A(x);   // path P
else
    x = expensive_B(x);   // path Q
```

At the source level this looks like two disjoint groups doing two independent things. On SIMT hardware there is no such thing as two groups running at once inside one warp — there is one program counter. So the hardware does the only thing it can: it runs **both** paths, one after the other, and uses a per-lane **active mask** to decide which lanes actually commit their results.

Concretely, the warp first executes path P with lanes 0–15 active and lanes 16–31 masked off (they go through the motions but their writes are discarded), then executes path Q with the masks reversed. The two halves never run simultaneously; they run **serially**. This is **warp divergence**, and its cost is not subtle: if both sides are equally expensive and the warp splits evenly, I pay for *both* and get useful work out of only half the lanes at a time. I have halved my throughput.[[sn: The masked-off lanes are not "free". They still occupy an issue slot and consume execution-unit cycles; they simply do not retire results. This is why divergence hurts throughput even though it does not corrupt correctness — the answer is right, it is just slow.]]

It gets worse with more targets. A `switch` with eight cases that scatters a warp across all eight arms can, in the worst case, serialize into eight passes, each with roughly 1/8 of the lanes doing useful work — close to a factor-of-8 haircut before any memory effect enters the picture.

The critical detail — the one that tripped me up the first time — is that **divergence only costs me when the branch splits threads within the same warp.** A branch every thread in a warp resolves the same way is free: the whole warp takes one side, no masking, no serialization. So `if (blockIdx.x == 0)` is usually fine (whole blocks, and thus whole warps, agree), while `if (threadIdx.x % 2 == 0)` is a disaster (it splits every warp down the middle).

## Reconvergence, and what changed at Volta

After a divergent region, the lanes need to get back in step — otherwise every warp would fragment further at each branch until the machine ran one lane at a time. The hardware brings them back together at a **reconvergence point**, historically the compiler-computed **immediate post-dominator** of the branch: the first instruction that *every* path is guaranteed to reach. The classic pre-Volta mental model was a hardware stack of `(mask, address)` entries — push the two branch targets, run one, run the other, pop back to the common continuation where the full mask is restored and all 32 lanes march forward again.

That stack model had a sharp edge. Because a diverged warp could only ever be executing *one* side at a time and could not interleave, threads on opposite sides of a branch could not make progress cooperatively. If I wrote a spin-lock where lane 0 holds the lock and lane 1 waits for it and put both in the same warp, pre-Volta I could **deadlock**: the waiting lane, being masked, never got to run, so the holding lane's release could never be observed. This was a real, documented footgun, and it burned me once.

**Volta introduced Independent Thread Scheduling** (ITS), and it survives on Turing, Ampere, Hopper, and Blackwell. The hardware now maintains a program counter *and* a call stack **per thread**, not per warp.[[sn: This is what the `.sync` suffix on Volta-era warp intrinsics is about — `__shfl_sync`, `__ballot_sync`, `__syncwarp`. Because lanes can now sit at different PCs, the compiler can no longer assume they are all present at a shuffle. I have to pass an explicit mask naming the lanes I expect to participate, and the hardware reconverges exactly those. The old maskless `__shfl` was deprecated for exactly this reason.]] The scheduler can now interleave execution between diverged groups of a warp — run a few instructions of path P, switch to path Q, come back — which means the spin-lock above no longer deadlocks: the waiter gets issue cycles, sees the release, and proceeds.

Here is the part that surprised me: ITS makes the model *safer* but not *faster*. Independent scheduling gives the hardware permission to interleave; it does **not** give it 32 program counters worth of execution width. There is still one instruction issued per cycle per warp, and diverged lanes still do not retire results in the same cycle. So the throughput math is unchanged — a split warp still under-utilizes the lanes — but the deadlock class of bugs is gone, and the compiler still opportunistically **reconverges** at the post-dominator to claw back efficiency whenever it can prove it is safe.[[sn: I can force it: `__syncwarp(mask)` is an explicit reconvergence barrier for the named lanes. Under ITS the compiler no longer guarantees threads reconverge at any particular point on their own, so if my algorithm depends on all lanes being lock-step at a spot — before a warp shuffle, say — I have to say so.]]

## Avoiding divergence in practice

The rule that falls out of all this is short: **make threads in the same warp agree.** I do not need branches to be globally uniform — I need them uniform *at warp granularity*, which is a much weaker and more achievable condition. A few patterns cover most of the cases I hit.

**Align branches to warp boundaries.** If a condition must differ across the grid, I arrange for it to differ *between* warps rather than within them. Branching on `threadIdx.x / 32` (the warp index within the block) never diverges — every lane in a warp gets the same quotient — whereas branching on `threadIdx.x % 2` diverges maximally. When I assign work, I prefer contiguous 32-thread chunks over interleaved ones.

**Turn small branches into arithmetic.** A short `if`/`else` that just selects a value is often better written branch-free, so there is no divergence to serialize at all:

```cpp
// Diverges: two masked passes if lanes disagree.
float y = (x > 0.0f) ? a : b;

// Branch-free: one pass, predicated select, no serialization.
float m = (float)(x > 0.0f);   // 1.0 or 0.0
float y = m * a + (1.0f - m) * b;
```

The compiler frequently does this for you — for a cheap two-sided select it emits a **predicated** instruction (a single instruction that each lane conditionally commits) rather than a real branch, which sidesteps divergence entirely.[[sn: In SASS you will see this as instructions guarded by a predicate register, e.g. `@P0 FADD ...`, instead of a `BRA` to two targets. Predication has no serialization cost — every lane issues the same instruction and the predicate just gates the write-back — but the compiler only chooses it when both arms are short. Long divergent regions still become real branches, which is why I hoist heavy work out of `if`s.]]

**Keep the boundary condition out of the hot loop.** The most common accidental divergence I create is the tail check — the `if (m < N && n < N)` guard I wrote in [the naive GEMM kernel](gemm-kernel-1-naive.html). It only diverges in the warps that straddle the matrix edge, so its total cost is small, but the principle generalizes: push ragged-edge handling to the boundary and let the interior run a single uniform path.

**Sort or bucket divergent work.** If different data items need genuinely different code paths — think a ray tracer where some rays hit and some miss — group the like items together *before* the kernel so that each warp processes 32 items that take the same path. This "stream compaction" or sorting step costs a pass over memory but can convert a 32-way-divergent kernel into a uniform one, and it is a standard move in production GPU code.

## The number, and the bridge

Put a figure on it. A kernel whose inner loop diverges evenly two ways does useful work in half its lanes; even with everything else perfect that is a hard ceiling of about **50% of the throughput** of the divergence-free version, and a wider `switch` scales that ceiling toward `1/k` for `k` equally-taken arms. Nsight Compute measures this directly as **warp execution efficiency** — the average fraction of active lanes per issued instruction. My good GEMM inner loop runs at essentially **100%**; the first time I saw a compute-bound kernel sitting at **40–60%**, that was the tell — a divergent branch was eating me alive, and no amount of tiling fixed it until the branch was gone.

This is the same discipline from [the three regimes](the-three-regimes.html): I predict the effect, then measure it. Before I profile, I read the kernel and ask which branches split a warp, predict the efficiency, then let Nsight confirm or embarrass me. Once I can see divergence, the SIMT model stops being an abstraction and becomes something I steer. Next I leave the execution model and follow the *data*: how those 32 lanes reach into memory together, why the hardware wants their addresses to line up, and how a single well-shaped access — [memory coalescing](gemm-kernel-2-coalescing.html) — took my GEMM ladder from a humiliating **1.3% of cuBLAS** to **8.5%** with a one-line change.
