A load from global memory costs on the order of **400 cycles**. That is not a typo and it is not a worst case — it is the ordinary, expected latency of reaching out to HBM and waiting for the bytes to come back. On a CPU that number would be a catastrophe; you would stall, and a fat out-of-order engine with branch prediction and a deep cache hierarchy would spend enormous transistor budget trying to paper over it. The **Streaming Multiprocessor** (SM) inside an H100 does almost none of that. It has no big reorder buffer, no speculative execution worth mentioning, and a laughably shallow pipeline by CPU standards. And yet it hits **989 TFLOP/s** of BF16 without breaking a sweat. The trick — the single most important idea in GPU architecture — is that the SM does not try to make any one thread fast. It keeps so many threads in flight that it never has to wait for any of them.

This article is about that trick: how the **warp scheduler** hides latency by switching among resident warps every cycle, why "many warps in flight" is the whole game, and how that idea turns directly into the occupancy number you stare at in Nsight.

## The hypothesis: hide latency, don't reduce it

When I first came to the GPU from the CPU world, this is the assumption I had to unlearn. The CPU strategy is *latency reduction* — make the memory access itself cheaper with caches, or reorder instructions so the pipeline keeps busy while one load is outstanding. The GPU strategy is the opposite: **latency hiding**. It accepts that a global load takes 400-ish cycles and simply arranges to always have other useful work to do during those cycles. My working hypothesis for this whole article: if I can keep enough warps resident, the 400-cycle stall never shows up in the wall-clock time at all.

The unit of scheduling that makes this work is the **warp** — a group of `32` threads that execute in lockstep, one instruction broadcast across all `32` lanes.[[sn: One warp is always exactly 32 threads on every NVIDIA GPU shipped to date; it is baked into the ISA, not a tunable. A thread block of 1024 threads is therefore 32 warps.]] Each SM is partitioned into four **processing blocks** (sometimes called SM sub-partitions), and each of those has its own warp scheduler and its own register file slice. Every cycle, each scheduler looks at the pool of warps assigned to it, finds one whose next instruction has all its operands ready, and issues that instruction. If the warp it issued last cycle is now stalled waiting on a load, fine — the scheduler just picks a different warp this cycle. No context save, no pipeline flush, no cost. The register file already holds every resident warp's state simultaneously, so a "context switch" is just choosing a different set of registers to read this cycle.

The picture above is the entire idea in one frame. There is exactly **one issue slot per scheduler per cycle**.[[sn: On Hopper each sub-partition can issue one instruction per cycle from one warp. Dual-issue from a single warp went away after the Kepler/Maxwell era; modern SMs get their throughput from *many warps*, not from issuing two instructions from one.]] The way you fill that slot on every single cycle is not by making one warp faster — it is by having enough warps around that at least one is always eligible.

## The napkin math: how many warps do you actually need?

This is where latency hiding stops being a vibe and becomes arithmetic. Ask the question directly: if a global load takes ~400 cycles, and one warp scheduler can issue one instruction per cycle, how many independent warps do I need to keep that scheduler busy for the whole 400 cycles?

If every warp issued one instruction and then immediately blocked on a 400-cycle load, you would need on the order of **400 warps** per scheduler to fully hide the latency — which is absurd, there aren't nearly that many. But real code doesn't work like that. Between two dependent loads a warp typically executes a handful of independent arithmetic instructions, and it is the *ratio* of stall latency to independent work that sets the requirement. This is just **Little's Law** wearing a GPU costume: the number of warps you need in flight equals the latency you're hiding divided by the throughput interval between the instructions that matter.

In practice the useful number is much smaller than 400 because (a) arithmetic instructions also have latency you overlap, and (b) each scheduler needs to cover its *own* latency, not the whole SM's. A common rule of thumb is that you want something like **a few dozen warps resident per SM** — enough that across the four schedulers, someone always has a ready instruction. The reason you don't need the full 400 is that arithmetic latency (an `FMA` is only a handful of cycles) is easy to hide, and memory latency is amortized across all the independent loads different warps have in flight at once.

## The direct link to occupancy

Everything above pays off in one word: **occupancy**. Occupancy is the ratio of warps you actually have resident on an SM to the hardware maximum. On the H100 an SM can hold up to **64 resident warps** (2048 threads), so 32 resident warps is 50% occupancy.[[sn: 64 warps is the architectural ceiling on Hopper. You rarely hit it, because registers or shared memory run out first — those are the real limiters, and occupancy is what falls out of them, not something you set directly.]] Occupancy is not a knob you turn. It is a *consequence* of how much of the SM's finite resources each of your blocks consumes, and there are three resources that fight over the same SM:

- **Registers.** The SM has a **256 KB register file** — `65536` 32-bit registers, shared across every resident thread. If each thread demands 64 registers, then `65536 / 64 = 1024` threads can be resident, i.e. 32 warps, i.e. 50% occupancy — and that is your ceiling no matter how many blocks you launch. Ask the compiler for 128 registers per thread and you have just halved your occupancy.[[sn: This is why the `-maxrregcount` flag and `__launch_bounds__` exist: they cap per-thread register usage so the compiler spills to local memory rather than shrinking occupancy. Sometimes that trade is worth it, sometimes the spills cost more than the extra warps buy — you profile to find out.]]
- **Shared memory.** The SMEM+L1 pool is **256 KiB per SM**, of which up to **228 KiB** is usable as shared memory. A block that asks for 100 KiB of SMEM means at most two such blocks fit per SM. If those blocks are small, you've capped occupancy through shared memory rather than registers.
- **Block/warp slots.** There are hard caps too — a maximum of `1024` threads per block, and a fixed number of resident blocks per SM.

Whichever of these runs out *first* sets your occupancy. The reason you care is exactly the latency-hiding story: fewer resident warps means fewer independent instructions for the scheduler to choose from, which means more cycles where every warp is stalled and the issue slot goes empty. That empty slot is exposed latency, and exposed latency is the number-one reason a memory-bound kernel crawls.

Here is the honest caveat that separates people who have read about occupancy from people who have profiled it: **more occupancy is not always better.** Past the point where you have enough warps to hide your latency, adding more warps buys you nothing — and it can actively hurt, because every extra resident warp is competing for the same L2 cache and the same register file. Kernels that carefully use *lots* of registers per thread to keep data on-chip (the good GEMM kernels do exactly this) deliberately run at modest occupancy because the register-resident data is worth more than the extra warps.[[sn: Volkov's famous "Better Performance at Lower Occupancy" is the canonical result here: a well-tiled GEMM at ~25-50% occupancy can beat the same kernel at 100% because instruction-level parallelism within each thread hides latency that would otherwise need more warps. Occupancy is a means, not the goal.]] Occupancy is a means to hide latency. If you're already hiding it, stop optimizing occupancy and go find the real bottleneck.

## Reading it in Nsight: the stall reasons

None of this is guesswork, because the profiler tells you exactly which warps are stalled and why. Point **Nsight Compute** (ncu) at a kernel and look at the **Warp State Statistics** section. It reports, for every issue slot, how many cycles warps spent eligible-but-not-issued versus stalled, and — the gold — it breaks the stalls down by *reason*. These reason names are the vocabulary you argue with the GPU in:

- **Stall Long Scoreboard** — the warp is waiting on a global/local memory load to come back. This is the 400-cycle stall from our figure. The "scoreboard" is the hardware that tracks outstanding memory operations; "long" means global/local memory, "short" means shared memory. Lots of Long Scoreboard means you are memory-latency bound and the cure is more warps to hide it, or fewer/cheaper loads.
- **Stall Short Scoreboard** — waiting on a shared-memory operation, which is far faster to satisfy than the long kind. But if you see a lot of it you probably have shared-memory bank conflicts serializing your accesses.
- **Stall MIO Throughput** — the memory input/output pipeline is saturated; too many warps hammering the same load/store or special-function units at once.
- **Stall Wait** — waiting on a short, fixed-latency instruction dependency (like an `FMA` result). Some of this is unavoidable; a pile of it means not enough independent work between dependent instructions — an ILP problem, not an occupancy problem.
- **Stall Not Selected** — the warp *was* ready, but the scheduler picked a different eligible warp this cycle. This one is counterintuitively *good* news: it means you have surplus eligible warps. If your dominant stall reason is "not selected," you are not latency-limited at all; you are issue-limited, and the answer is not more warps.

The discipline is the same predict-then-measure loop from [the three regimes](the-three-regimes.html): before you run ncu, say out loud what you expect the dominant stall reason to be. "This kernel does one dependent global load per iteration with almost no arithmetic between loads, so I expect it to be dominated by Long Scoreboard and I expect low achieved occupancy to be exposing it." Then check. When your prediction matches, you understand the kernel. When it doesn't, you've found something — a bank conflict you didn't expect, a register spill quietly capping your warps.

## What this buys us on the ladder

Latency hiding is the mechanism underneath every number in our GEMM climb. The [naive kernel](gemm-kernel-1-naive.html) sits at a humiliating **1.3% of cuBLAS** precisely because it issues a fresh global load for every element with almost no independent work between loads — the scheduler runs out of ready warps and the issue slot goes empty, cycle after cycle. Nsight would show it pinned on **Stall Long Scoreboard**. The whole rest of the ladder is, from this angle, a campaign to give the warp scheduler more to do during those 400-cycle windows: coalescing makes each load cheaper, [shared memory](shared-memory-l1.html) turns hundreds of slow global loads into a handful of fast on-chip ones, and register tiling packs enough independent `FMA`s between memory ops that a modest number of warps hides all the latency there is to hide. By the time we reach the warp-tiled kernel at **93.7% of cuBLAS**, we are not running at high occupancy at all — we are running with just enough warps, each doing a lot of independent work, and a scheduler whose issue slot is full almost every cycle.

That is the real definition of a fast kernel: not one that computes quickly, but one that never waits. In the next section we make the loads themselves cheaper, so that every cycle the scheduler does spend issuing is spent moving as many useful bytes as the memory system will give us.
